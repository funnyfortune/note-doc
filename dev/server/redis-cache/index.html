缓存

说明

在实际生产中，往往随着数据增多，存在冷热数据，数据库查询过慢，导致响应效率下降，体现性不好的情况。这时候，我们会先考虑对热数据进行缓存处理。 当我们使用缓存时，目标通常有两个：第一，提升响应效率和并发量；第二，减轻数据库的压力。

在项目使用缓存的方式，有如下两种方式：

  • 编程式缓存：基于 Spring Data Redis 框架的 RedisTemplate 操作模板

  • 声明式缓存：基于 Spring Cache 框架的 @Cacheable 等等注解

# 缓存常见问题

# 缓存穿透

缓存穿透（cache penetration）是用户访问的数据既不在缓存当中，也不在数据库中。出于容错的考虑，如果从底层数据库查询不到数据，则不写入缓存。这就导致每次请求都会到底层数据库进行查询，缓存也失去了意义。当高并发或有人利用不存在的Key频繁攻击时，数据库的压力骤增，甚至崩溃，这就是缓存穿透问题。

解决方案：缓存 NULL 数据

# 缓存击穿

缓存雪崩是指只大量热点 key 同时失效的情况，如果是单个热点 key，在不停的扛着大并发，在这个 key 失效的瞬间，持续的大并发请求就会击破缓存，直接请求到数据库，好像蛮力击穿一样。这种情况就是缓存击穿（Cache Breakdown）。

解决方案：加锁

# 缓存雪崩

在使用缓存时，通常会对缓存设置过期时间，一方面目的是保持缓存与数据库数据的一致性，另一方面是减少冷缓存占用过多的内存空间。

但当缓存中大量热点缓存采用了相同的实效时间，就会导致缓存在某一个时刻同时实效，请求全部转发到数据库，从而导致数据库压力骤增，甚至宕机。从而形成一系列的连锁反应，造成系统崩溃等情况，这就是缓存雪崩（Cache Avalanche）。

解决方案：所有key都添加上随机的过期时间

友情提示：

如果需要了解 Spring Cache 框架，可以阅读 请查看官网 (opens new window) 文档。

# Cache 和 CacheManager 接口说明

Cache 接口包含缓存的各种操作集合，你操作缓存就是通过这个接口来操作的。

Cache 接口下 Spring 提供了各种 xxxCache 的实现，比如：RedisCache、EhCache、ConcurrentMapCache

CacheManager 定义了创建、配置、获取、管理和控制多个唯一命名的 Cache。这些 Cache 存在于 CacheManager 的上下文中。

CacheResolver 是 Cache 处理的拦截器，通过实现该接口，对缓存进行拦截处理。

# 注解详解

# @Cacheable

@Cacheable (opens new window) 注解：添加在方法上，缓存方法的执行结果。执行过程如下：

  • 1）首先，判断方法执行结果的缓存。如果有，则直接返回该缓存结果。

  • 2）然后，执行方法，获得方法结果。

  • 3）之后，根据是否满足缓存的条件。如果满足，则缓存方法结果到缓存。

  • 4）最后，返回方法结果。

属性说明

cacheNames/value ：用来指定缓存组件的名字

key ：缓存数据时使用的 key，可以用它来指定。默认是使用方法参数的值。（这个 key 你可以使用 spEL 表达式来编写）

keyGenerator ：key 的生成器。 key 和 keyGenerator 二选一使用

cacheManager ：可以用来指定缓存管理器。从哪个缓存管理器里面获取缓存。

condition ：可以用来指定符合条件的情况下才缓存

unless ：否定缓存。当 unless 指定的条件为 true ，方法的返回值就不会被缓存。当然你也可以获取到结果进行判断。（通过 #result 获取方法结果）

sync ：是否使用异步模式。

# @CacheEvict 注解

@CacheEvict (opens new window) 使用了 @CacheEvict 注解的方法，会清空指定缓存。 如下示例：

@CacheEvict(value = "test_cache", key="#ids")
public void delete(List<Long> ids){
}
 // 同时删除多个缓存方法：
@Caching(evict = {
@CacheEvict(value =  "test_cache", key="#ids"),
@CacheEvict( value = "test_cache1", key="#ids")
})
// 删除所有
@CacheEvict(value = {"test_cache", "test_cache1"},allEntries = true)

@CacheConfig (opens new window) 该缓存作用在类上， 可以通过指定 cacheNames/value 等属性的策略，为 @Cacheable， @CacheEvict ... 这些方法注解提供统一的缓存策略； 若上述方法的注解有指定，则会被覆盖。

# TTL 设置

# 1. 全局配置

Spring Cache 提供配置方式，为缓存设置超时时间，若不显式设置，则永不过期。 如下示例：

spring:
  cache:
    type: redis
    redis:
      time-to-live: 3m # 3分钟

# Spring Cache 不足

SpringCache

  1. 只对读模式下的缓存失效进行了处理，对于写模式下的缓存失效没有相应的处理，需要我们自己采取其他方式来处理。
  2. 缓存的类型只有 String， 若需要其它类型存储，则需要采用编程式缓存。 缓存中常见的失效场景及解决方案：

使用建议

  1. 对于常规数据（读多写少，及时性、一致性要求不高的数据）完全可以使用 Spring Cache。
  2. 对于特殊数据（比如要求高一致性）则需要特殊处理。

缓存穿透：cache-null-values: true，允许写入空值 缓存击穿：@Cacheable(sync = true)，加锁 缓存雪崩：time-to-live:xxx，设置不同的过期时间

# 编程式缓存使用

# RedisService

RedisService 通过对 RedisTemplete 进行封装，在需要使用的类上注入即可。 示例如下

    /**
     * 添加hashset,并且设置存活时间
     * @TC
     */
    public <T> Integer hSet(final IRedisKey key, final String field, final T value,
                            final long liveTime) {
        return redisTemplate.execute((RedisCallback<Integer>) connection -> {
            connection.select(key.getDb());
            connection.hSet(getKeyBytes(key), field.getBytes(), serializer.serialize(value));
            if (liveTime > 0) {
                connection.expire(getKeyBytes(key), liveTime);
            }
            return 0;
        });
    }

# 缓存刷新

# 方式一：定时刷新

通过定时任务调度，该方式适合要求实时性低的数据刷新。例如：token 刷新、分析统计数据、系统配置参数等。 另外任务调度刷新，具有对缓存数据的容错性，保障了数据能缓存起来。 示例如下：

  1. 在业务模块编写缓存刷新接口。
  2. 在 myapp-job-ecutor 模块中，建立 handler，调用微服务接口。
   /**
    * token
    */
   @XxlJob("tokenJobHandler")
   public void tokenJobHandler() throws Exception {
       TokenDTO dto = new TokenDTO();
       dto.setAccount(appid);
       dto.setPassword(secret);
       R<TokenVO> res = tokenServiceApi.getToken(dto);
       if(res == null || res.getCode() != R.ok().getCode()){
           XxlJobHelper.handleFail("获取token异常!"+res);
           return;
       }
       TOKEN = res.getData().getToken();
   }
